## 执行摘要

Superpowers 是一套面向 AI 编码智能体（尤其是 Claude Code）的**完整软件开发工作流框架**，通过 15+ 个可组合技能（Skills）系统性地重塑 AI 代理的开发行为——强制"设计先于编码"、"测试先于实现"、"验证先于声明完成"。它的核心价值在于**将结构化工程纪律以技能文件的形式注入 AI 智能体行为**，解决了 LLM 默认行为中最常见的工程质量陷阱。

---

## 一、核心方法论

### 1.1 解决的根本问题

AI 编码智能体存在三类典型质量问题：

| 问题类型 | 典型表现 | 危害 |
|----------|----------|------|
| **急于实现** | 收到需求立刻写代码，跳过设计 | 返工率高、需求偏差大 |
| **跳过测试** | 手动测试代替 TDD，事后补测试 | 测试无效、回归难发现 |
| **虚假完成** | "应该可以了"而未实际运行验证 | 破损代码进入主干 |
| **随机调试** | 多处同时修改尝试排查问题 | 掩盖根因，制造新 bug |
| **上下文污染** | 单一大会话处理所有任务 | 前期错误干扰后期判断 |

Superpowers 对上述每一个问题都有对应的强制性技能规则（rigid skills）。

### 1.2 核心概念矩阵

| 概念 | 定义 | 关键约束 |
|------|------|----------|
| **Skills（技能）** | 定义智能体行为的 Markdown 文件，覆盖特定工作阶段 | 高优先级，覆盖默认系统提示 |
| **Rigid Skills（刚性技能）** | 必须严格遵循的技能（TDD、调试） | 无例外，无适配 |
| **Flexible Skills（弹性技能）** | 可根据场景调整的技能（规划、协作） | 遵循原则，允许上下文调整 |
| **Subagent（子智能体）** | 接受独立任务的新鲜会话智能体 | 每任务独立，无历史上下文污染 |
| **Worktree（工作树）** | Git 隔离工作区，一个特性一个分支 | 开发与主干隔离 |
| **Design Gate（设计门控）** | 阻止实现动作直到设计被审批 | 硬性阻断，无绕过 |

### 1.3 架构概览

```
┌─────────────────────────────────────────────────────────────────┐
│                    Superpowers 工作流架构                         │
│                                                                 │
│  ① Brainstorming  →  ② Git Worktrees  →  ③ Writing Plans      │
│     (设计门控)          (隔离工作区)         (任务分解)             │
│         │                   │                    │             │
│         ↓                   ↓                    ↓             │
│  ④ Subagent-Driven  →  ⑤ TDD Execution  →  ⑥ Code Review    │
│   Development           (RED-GREEN-REFACTOR)   (两阶段审查)       │
│     (每任务独立子智能体)                              │             │
│                                                    ↓             │
│                                           ⑦ Finishing Branch   │
│                                              (验证+合并决策)      │
└─────────────────────────────────────────────────────────────────┘

         辅助技能（可随时调用）
         ├── systematic-debugging（四阶段根因调试）
         ├── dispatching-parallel-agents（并行独立任务）
         ├── verification-before-completion（完成声明前验证）
         └── using-superpowers（技能调用元规则）
```

**技能优先级层次**：
```
1. 用户明确指令         ← 最高优先级
2. Superpowers Skills  ← 覆盖默认提示词行为
3. 默认系统提示         ← 最低优先级
```

### 1.4 决策与交互模式

#### 设计门控（Brainstorming）
```
收到需求
    ↓
阅读项目上下文
    ↓
逐一提出澄清问题
    ↓
提出 2-3 种方案（含权衡分析）
    ↓
分段呈现设计（每段等待审批）
    ↓
写入设计文档 docs/superpowers/specs/YYYY-MM-DD-xxx.md
    ↓
子智能体审查规格（最多3轮）
    ↓
用户最终审批 ← [无此步骤则绝不进入实施]
    ↓
调用 writing-plans 技能
```

#### 子智能体驱动开发（Subagent-Driven Development）
```
从计划文档提取所有任务（TodoWrite）
    ↓
针对每个任务：
    ├── 派发独立子智能体（携带完整上下文，无需子智能体读文件）
    ├── 子智能体实施
    ├── 规格合规性审查（第一阶段）
    ├── 代码质量审查（第二阶段）
    └── 如有问题：实施者修复 → 重新审查（循环直至通过）
    ↓
所有任务完成后做最终代码审查
```

#### 并行智能体调度（Dispatching Parallel Agents）
```
识别独立故障域（≥3个，互不影响）
    ↓
验证独立性（修复A不会自动修复B）
    ↓
每个域创建独立的、自包含的子智能体任务
    ↓
并发派发（同时不是顺序）
    ↓
收集结果 → 检查冲突 → 运行完整测试套件
```

## 二、可复用模式提取

### 2.1 设计模式详解

| 模式名称 | 描述 | 最佳应用场景 |
|----------|------|-------------|
| **设计门控模式** | 在设计文档被审批前硬性阻断所有实施行动 | 所有功能开发任务，尤其复杂需求 |
| **两阶段审查模式** | 先审规格合规，再审代码质量，顺序不可颠倒 | 每个实施任务完成后 |
| **新鲜子智能体模式** | 每个任务派发无历史污染的新会话子智能体 | 独立性强的任务清单执行 |
| **并行域分割模式** | 独立问题域并发处理，共享状态场景禁用 | 多文件独立bug修复、并行功能开发 |
| **隔离工作树模式** | 每个特性在独立 git worktree 中开发 | 需要干净测试基线的所有开发 |
| **强制 TDD 循环** | 先写失败测试，再写最少代码通过，最后重构 | 所有生产代码编写 |
| **四阶段根因调试** | 根因调查→模式分析→假设测试→实施，无跳过 | 任何 bug 修复场景 |
| **完成前验证模式** | 声明完成前必须运行命令并展示真实输出 | 任务完成汇报时 |
| **技能优先检查** | 执行任何操作前先检查是否有适用技能 | 每次开始新任务时 |
| **计划任务粒度规范** | 每个任务 2-5 分钟，含精确文件路径和命令 | 编写实施计划时 |

### 2.2 反模式识别

| 反模式 | 为何有问题 | 正确做法 |
|--------|------------|----------|
| **收到需求立即写代码** | 需求理解偏差导致大量返工 | 先完成 brainstorming 设计门控 |
| **测试后置** | 后写的测试立即通过，无法验证正确性 | 严格 RED-GREEN-REFACTOR，代码在测试之前必须删除 |
| **"应该可以了"声明完成** | 假设代替证据，破损代码进主干 | 运行命令，展示完整输出，才能声明完成 |
| **同时修改多处调试** | 无法隔离变量，掩盖根因 | 每次假设只做最小改动 |
| **3次修复失败仍继续** | 策略已证明无效 | 停止并质疑架构本身 |
| **跳过技能直接行动** | 失去结构化流程保护 | "1%概率适用就必须调用" |
| **单一会话处理所有任务** | 历史上下文污染后续判断 | 每个实施任务派发新鲜子智能体 |
| **并行处理有依赖关系的任务** | 智能体间状态干扰 | 严格验证独立性才能并行 |
| **大型整体文件** | 难以测试、责任不清 | 按职责拆分，单一职责原则 |
| **简单项目跳过设计** | "连 todo list 也需要文档化设计" | 无论规模均需设计文档 |

## 三、业务应用分析

### 3.1 直接适用场景

**场景一：企业内部 AI 编码助手标准化**
将 Superpowers Skills 部署为团队级 Claude Code 配置，强制所有 AI 辅助开发遵循统一工程规范。无需改变开发者习惯，由技能系统兜底保证质量。

**场景二：复杂软件项目自动化开发流水线**
使用 brainstorming → writing-plans → subagent-driven-development 完整链路处理中等复杂度特性开发，减少人工介入点。

**场景三：遗留系统 Bug 修复自动化**
systematic-debugging 四阶段流程特别适合不熟悉的遗留代码库——强制根因调查而非猜测式修复，降低引入新 bug 风险。

**场景四：并行 A/B 方案探索**
dispatching-parallel-agents 模式适合同时探索多个技术方案（如性能优化方向、重构策略选择），缩短方案验证周期。

**场景五：AI 代码生成质量保障**
TDD 技能 + verification-before-completion 技能组合，可作为 AI 生成代码的质量门控，集成到 CI/CD 前的人工审查节点。

### 3.2 投资回报分析

| 维度 | 评估 | 说明 |
|------|------|------|
| **实施复杂度** | 低 | 仅需复制技能文件到项目 `.claude/skills/` 目录 |
| **学习曲线** | 低-中 | 理解技能优先级体系需要约 2-4 小时 |
| **收益时间** | 快（1-2周） | 技能立即生效，效果在首个项目迭代可见 |
| **主要收益** | 高 | 减少设计-编码偏差、TDD 覆盖率提升、调试效率提升 |
| **适用团队规模** | 个人 → 企业 | 从个人开发者到大型团队均可 |
| **维护成本** | 极低 | 技能文件是静态 Markdown，无运行时依赖 |

## 四、实施路线图

### 阶段 1：基础接入（0.5 天）

1. 克隆或下载 Superpowers 技能文件到本地项目
2. 在项目根目录创建 `.claude/skills/` 目录结构
3. 复制核心技能：`using-superpowers`、`brainstorming`、`test-driven-development`
4. 在 CLAUDE.md 中添加技能调用规则说明
5. 在一个小型真实任务中运行完整的 brainstorming → TDD 流程验证

### 阶段 2：完整工作流集成（1-3 天）

1. 添加剩余核心技能：`writing-plans`、`subagent-driven-development`、`systematic-debugging`
2. 配置 `verification-before-completion` 作为任务完成检查点
3. 为团队编写技能使用入门文档（对应你项目的具体规范）
4. 在中等复杂度特性（2-5天工作量）上运行完整工作流，收集反馈
5. 根据项目特性调整 `writing-plans` 的任务粒度规范

### 阶段 3：高级模式与定制（1 周）

1. 引入 `using-git-worktrees` + `dispatching-parallel-agents` 支持并行开发
2. 定制 `code-reviewer` agent 的审查维度，对齐团队代码标准
3. 基于 `writing-skills` 技能，为项目特定领域创建自定义技能
4. 建立技能效果度量指标（设计偏差率、TDD 覆盖率、调试周期）
5. 收集团队反馈，持续迭代技能文件内容

## 五、关键洞察总结

### 洞察 1：技能 = 强制性工程纪律的外化

Superpowers 最核心的创新不是工具，而是**将工程纪律从"最佳实践建议"升级为"智能体必须执行的刚性约束"**。`using-superpowers` 技能明确规定："如果你认为有 1% 的概率某个技能适用，你绝对必须调用它"。这解决了 AI 智能体天然倾向于走捷径的问题。

### 洞察 2：新鲜子智能体解决上下文污染

传统单会话开发中，早期的错误推断会污染后续所有决策。**每任务一个新鲜子智能体**（携带完整的书面上下文而非历史对话）彻底隔离了这种污染，代价仅是需要写清楚的任务规格——这反而倒逼了更好的规格质量。

### 洞察 3：两阶段审查的顺序锁定价值

先审规格合规（"做了正确的事吗？"），再审代码质量（"做得好吗？"）——顺序至关重要。如果顺序颠倒，会导致在方向错误的代码上投入质量优化资源，最终仍需大规模返工。

### 洞察 4："简单任务也需要设计"反直觉价值

Superpowers 明确反对"这个任务太简单了，不需要设计"的思维——这是最常见的规范跳过借口。实践中，正是"简单"任务积累了大量隐性技术债。

### 洞察 5：验证文化的系统性建立

`verification-before-completion` 技能直接对抗 AI 智能体最危险的行为模式：自信声称已完成但未经实际验证。通过要求展示真实命令输出作为完成证据，建立了一种可量化、可审计的完成标准。
