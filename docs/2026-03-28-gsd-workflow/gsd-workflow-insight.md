# GSD (Get Shit Done) - 深度洞察报告

## 执行摘要

GSD 是一个轻量级但强大的元提示和上下文工程系统，专门设计用于解决 AI 辅助编程中的**上下文衰减问题**（context rot）——即随着上下文窗口填充而导致的质量下降现象。通过系统化的项目初始化、阶段规划、并行执行和验证流程，GSD 将 Claude Code 从"生成不一致代码"的工具转变为"可可靠地构建复杂系统"的工作流。

**核心价值**：将复杂性封装在系统内部，用户只需简单命令即可获得企业级的开发质量。

---

## 一、核心方法论

### 1.1 解决的根本问题

**上下文衰减（Context Rot）**：当 AI 的上下文窗口逐渐被填充时，输出质量会下降。具体表现为：
- "我现在会更简洁"（暗示容量不足）
- 遗忘早期讨论的内容
- 决策不一致
- 代码质量下降

**Vibecoding 的信誉危机**：
- 用户描述需求 → AI 生成代码 → 得到不一致的垃圾
- 无法规模化
- 缺乏全局理解

**现有工具的过度复杂化**：
- 引入 sprint 仪式、story points、利益相关者同步、回顾会议、Jira 工作流
- 假装自己是 50 人软件公司
- 实际需求：简单、直接、有效

### 1.2 核心概念矩阵

| 概念 | 定义 | 作用 |
|------|------|------|
| **元提示（Meta-prompting）** | 系统生成结构化提示，而非用户直接输入 | 确保 AI 收到精确指令，避免歧义 |
| **上下文工程（Context Engineering）** | 管理哪些信息在何时进入 AI 上下文 | 控制上下文窗口使用，保持质量稳定 |
| **规范驱动开发（Spec-driven Development）** | 从需求→规划→执行的严格流程 | 确保交付符合预期，可验证 |
| **阶段（Phase）** | 里程碑中的一组相关功能 | 将大项目分解为可管理的块 |
| **计划（Plan）** | 原子任务，包含 XML 结构和验证步骤 | 可在新的上下文窗口中独立执行 |
| **波次（Wave）** | 基于依赖关系的并行执行组 | 最大化并行度，最小化等待时间 |
| **工作流（Workflow）** | discuss → plan → execute → verify 的循环 | 每个阶段都经过充分研究、规划和验证 |
| **里程碑（Milestone）** | 版本周期（v1, v2, ...） | 清晰的发布节奏和范围管理 |

### 1.4 决策与交互模式

**核心工作流循环**：
```
new-project → discuss-phase → plan-phase → execute-phase → verify-work → ship
      ↑                                                                              ↓
      └─────────────────────── complete-milestone ←──────────────────────────────────┘
                                      ↓
                                new-milestone
```

**1. 项目初始化模式**：
```
用户输入 → 问题收集（直到完全理解）→ 并行研究（可选）→ 需求提取 → 路线图生成 → 用户批准
```
- 输出：`PROJECT.md`, `REQUIREMENTS.md`, `ROADMAP.md`, `STATE.md`, `research/`

**2. 阶段讨论模式**（Discuss Phase）：
```
分析阶段 → 识别灰色领域 → 针对性提问 → 生成 CONTEXT.md
```
- 灰色领域类型：
  - **视觉特性**：布局、密度、交互、空状态
  - **API/CLI**：响应格式、标志、错误处理、详细程度
  - **内容系统**：结构、语调、深度、流程
  - **组织任务**：分组标准、命名、重复项、异常

**3. 阶段规划模式**（Plan Phase）：
```
研究（如何实现）→ 规划（2-3 个原子计划）→ 验证（符合需求）→ 迭代（直到通过）
```
- 每个计划足够小，可在新的上下文窗口中执行
- XML 结构确保精确指令

**4. 波次执行模式**（Execute Phase）：
```
计划分组（基于依赖）→ 波次内并行 → 波次间串行 → 原子提交 → 目标验证
```

**5. 验证模式**（Verify Work）：
```
提取可测试交付物 → 逐个引导用户 → 失败自动诊断 → 生成修复计划
```

## 二、提取的模式

### 2.1 设计模式详解

| 模式 | 描述 | 何时使用 | 实现要点 |
|------|------|----------|----------|
| **上下文分层加载** | 根据阶段动态加载相关上下文，避免一次性加载所有信息 | 大型项目，避免上下文溢出 | • PROJECT.md 始终加载<br>• 阶段相关文档按需加载<br>• 控制单次加载大小 |
| **XML 结构化提示** | 使用 XML 格式定义任务，包含明确指令和验证步骤 | 需要精确执行的任务 | • `<task>` 根元素<br>• `<name>` 任务名称<br>• `<files>` 目标文件<br>• `<action>` 执行指令<br>• `<verify>` 验证命令<br>• `<done>` 完成标准 |
| **波次并行执行** | 基于依赖关系将计划分组，波次内并行，波次间串行 | 有多个独立任务的阶段 | • 分析依赖关系<br>• 无依赖 → 同一波次<br>• 有依赖 → 后续波次<br>• 垂直切片比水平切片更好并行化 |
| **原子 Git 提交** | 每个任务完成后立即独立提交 | 需要清晰历史和可回滚性的项目 | • 一个任务 = 一个提交<br>• 提交消息：`type(scope): description`<br>• 支持 git bisect<br>• 独立回滚 |
| **编排者-子智能体** | 轻量级编排者协调专业化子智能体 | 复杂工作流，需要专业化能力 | • 编排者不承担繁重工作<br>• 生成子智能体并等待结果<br>• 主上下文保持 30-40% 使用率<br>• 子智能体在全新上下文中工作 |
| **假设模式 vs 讨论模式** | 讨论模式：提问获取用户偏好；假设模式：分析代码库提出假设，用户纠正 | 代码库完善时用假设模式，全新项目用讨论模式 | • 讨论模式：适合新项目<br>• 假设模式：适合现有代码库<br>• 可通过 workflow.discuss_mode 配置 |
| **里程碑驱动开发** | 按里程碑组织（v1, v2, ...），每个里程碑包含多个阶段 | 长期项目，需要清晰发布节奏 | • 每个里程碑：定义 → 构建 → 发布<br>• 完成后归档并标记版本<br>• 新里程碑重新开始循环 |
| **快速模式** | 对于临时任务跳过可选步骤，保留核心保证（原子提交、状态跟踪） | 小任务、快速修复 | • 相同智能体（规划器 + 执行器）<br>• 跳过研究、计划检查、验证器<br>• 独立跟踪：`.planning/quick/` |
| **种子想法（Seeds）** | 捕获前瞻性想法，在正确的里程碑自动浮现 | 有未来想法但不希望现在实现 | • 带触发条件存储<br>• 自动在适当里程碑提示<br>• 防止想法丢失 |
| **工作流（Workstreams）** | 命名空间隔离的并行里程碑工作 | 需要同时处理多个功能分支 | • 创建隔离工作流<br>• 切换活动工作流<br>• 完成后合并 |

### 2.2 反模式识别

| 反模式 | 为何有问题 | 替代方案 | GSD 的解决方案 |
|--------|------------|----------|----------------|
| **企业剧院式复杂化** | 引入不必要的仪式（sprint、story points、Jira） | 保持简单，直接有效 | GSD 将复杂性封装在系统内部，用户只需几个命令 |
| **一次性加载所有上下文** | 导致上下文衰减，质量下降 | 分层加载，按需提供 | 上下文分层加载：全局（PROJECT.md）+ 阶段（CONTEXT.md）+ 计划（PLAN.md） |
| **模糊的自然语言指令** | AI 猜测意图，结果不一致 | 结构化、精确指令 | XML 结构化提示：明确的 action、verify、done |
| **顺序执行所有任务** | 浪费时间，独立任务可以并行 | 分析依赖关系，并行执行独立任务 | 波次并行执行：依赖分析 → 波次分组 → 并行执行 |
| **批量提交所有更改** | 历史不清晰，难以回滚，无法定位问题 | 原子提交，每个任务独立提交 | 嚼子 Git 提交：一个任务 = 一个提交 |
| **主智能体承担所有工作** | 上下文窗口填充，质量下降 | 专业子智能体，编排协调 | 编排者-子智能体：主上下文保持 30-40%，子智能体在全新上下文工作 |
| **缺少验证步骤** | 功能可能不符合预期 | 执行后验证 | 验证模式：提取可测试交付物 → 用户测试 → 失败诊断 → 自动修复 |
| **水平切片并行化** | 层间依赖多，难以真正并行 | 垂直切片，端到端功能 | 波次分组鼓励垂直切片（Plan 1: 用户端到端，Plan 2: 产品端到端） |
| **忽略现有代码库** | 新功能不符合现有模式 | 先分析代码库 | `/gsd:map-codebase` → 并行智能体分析栈、架构、约定、关注点 |
| **全自动驾驶** | 用户失去控制，结果可能不符合预期 | 关键决策点确认 | 模式配置：interactive（确认每个步骤）vs yolo（自动批准） |

## 三、业务应用分析

### 3.1 直接适用场景

| 场景 | 适用性 | 说明 |
|------|--------|------|
| ** solo 开发者/独立黑客** | ⭐⭐⭐⭐⭐ | 核心目标用户：一个人想快速构建复杂产品，无需假装是大公司 |
| ** AI 辅助开发团队** | ⭐⭐⭐⭐⭐ | 任何使用 Claude Code/OpenCode/Gemini CLI 的团队，提高质量和一致性 |
| ** 原型到产品** | ⭐⭐⭐⭐⭐ | 从想法到可工作产品的快速迭代，每个里程碑都是可发布版本 |
| ** 遗留系统现代化** | ⭐⭐⭐⭐ | 使用 `/gsd:map-codebase` 先分析现有代码，然后系统化地添加新功能 |
| ** 多并行项目** | ⭐⭐⭐⭐ | 使用工作流（workstreams）管理多个功能分支 |
| ** 契约驱动外包** | ⭐⭐⭐⭐ | 明确的需求、计划和验收标准，适合外包或团队协作 |
| ** 教育和培训** | ⭐⭐⭐ | 学习如何系统地使用 AI 进行软件开发，理解上下文工程 |

### 3.2 适配需求

**技术栈要求**：
- **最低要求**：使用支持的 AI 运行时（Claude Code, OpenCode, Gemini CLI, Codex, Copilot, Cursor, Windsurf, Antigravity）
- **推荐**：Node.js 环境（用于安装 GSD）
- **建议**：使用 `--dangerously-skip-permissions` 模式或配置细粒度权限，以避免频繁确认

**团队协作适配**：
- **小团队（2-5 人）**：直接使用，每人一个工作流
- **中型团队（5-20 人）**：需要一个"集成者"角色，协调不同工作流的合并
- **大团队（20+ 人）**：需要额外的流程层（代码审查、CI/CD 集成），GSD 可以作为个体生产力工具

**非软件开发适配**：
GSD 的核心理念（上下文工程、阶段化规划、原子执行）可以适配到其他领域：
- **技术写作**：研究 → 大纲 → 草稿 → 审查
- **数据分析**：需求收集 → 数据准备 → 分析 → 可视化
- **产品设计**：用户研究 → 概念设计 → 原型 → 测试

### 3.3 投入与回报

| 维度 | 评估 | 说明 |
|------|------|------|
| **学习曲线** | 中等（1-2 周熟练） | 核心命令简单（5 个核心命令），但高级功能需要时间 |
| **初始化时间** | 低（5-10 分钟） | `npx get-shit-done-cc@latest` 一键安装 |
| **项目启动时间** | 中等（30-60 分钟） | `/gsd:new-project` 需要问答、研究、规划 |
| **每阶段时间** | 低到中等 | 简单阶段：几分钟；复杂阶段：几小时（主要是研究） |
| **资源消耗** | 中等（Token 成本） | 取决于模型配置（quality/balanced/budget）和项目复杂度 |
| **维护负担** | 低 | 系统自我管理，主要是更新（`npx get-shit-done-cc@latest`） |
| **预期收益** | 高 | • 一致的代码质量<br>• 清晰的 git 历史<br>• 可验证的功能<br>• 更快的迭代速度 |

**ROI 分析**：
- **一次性投入**：安装和学习（1-2 天）
- **持续投入**：每项目的初始化（1 小时）+ 每阶段的讨论和验证（变化）
- **回报**：
  - 减少 50-80% 的手动编码时间（由 AI 完成）
  - 减少 70-90% 的调试时间（原子提交 + 验证）
  - 减少 90% 的"我不知道上次做到哪了"时间（STATE.md 记忆）
  - 提高代码质量和一致性（结构化提示 + 验证）

## 四、实施路线图

### 阶段 1：基础建设（1-2 周）

**目标**：安装 GSD，理解核心概念，完成第一个小项目

**步骤**：
1. **安装 GSD**
   ```bash
   npx get-shit-done-cc@latest
   # 选择：Claude Code, Global
   ```

2. **配置权限**
   ```bash
   # 方案 A：推荐（跳过权限模式）
   claude --dangerously-skip-permissions

   # 方案 B：细粒度权限
   # 编辑 ~/.claude/settings.json 或 ./.claude/settings.json
   # 添加 Bash 命令白名单（见 README）
   ```

3. **学习核心命令**
   - `/gsd:help` - 查看所有命令
   - 阅读 README 的 "Why It Works" 部分
   - 理解：上下文工程、XML 提示、波次执行、原子提交

4. **完成第一个项目**（选择小型项目）
   - 创建一个简单的 CLI 工具（如 todo list）
   - 或添加一个小功能到现有项目
   - 走完整流程：new-project → discuss → plan → execute → verify

**预期输出**：
- 熟悉 5 个核心命令
- 理解 `.planning/` 目录结构
- 体验一次完整的 GSD 循环

### 阶段 2：核心实施（2-4 周）

**目标**：在真实项目中使用 GSD，掌握高级功能

**步骤**：
1. **现有项目分析**（如果已有代码库）
   ```bash
   /gsd:map-codebase
   ```
   - 等待并行智能体完成分析
   - 查看 `.planning/codebase/` 中的报告

2. **启动新里程碑**
   ```bash
   /gsd:new-milestone
   ```
   - 描述要构建的下一个功能
   - 系统生成新的 ROADMAP.md

3. **配置模型配置文件**
   ```bash
   /gsd:settings
   # 选择模型配置：quality（高质量） / balanced（平衡） / budget（经济）
   ```

4. **自定义工作流**
   - 编辑 `.planning/config.json`
   - 切换讨论模式：`workflow.discuss_mode = 'assumptions'`（如果代码库完善）
   - 启用/禁用可选智能体：`workflow.plan_check`, `workflow.verifier`

5. **尝试高级功能**
   - `/gsd:quick` - 快速任务
   - `/gsd:workstreams create <name>` - 创建工作流
   - `/gsd:plant-seed <idea>` - 种子想法

**预期输出**：
- 完成至少一个完整的里程碑
- 理解如何调整 GSD 以适应工作风格
- 掌握至少 3 个高级功能

### 阶段 3：集成与优化（持续）

**目标**：将 GSD 深度集成到开发工作流中，优化效率

**步骤**：
1. **Git 集成**

## 五、关键洞察总结

### 技术洞察

1. **上下文工程是 AI 辅助编程的关键**
   - 问题：上下文窗口填充 → 质量下降
   - 解决：分层加载、按需提供、结构化提示

2. **编排者-子智能体模式保持主上下文清洁**
   - 主会话保持 30-40% 使用率
   - 子智能体在全新 200k token 上下文中工作
   - 结果：一致的高质量输出

3. **原子 Git 提交是 AI 编程的最佳实践**
   - 可追溯：每个任务独立可识别
   - 可回滚：独立回退不影响其他任务
   - 可调试：git bisect 精确定位问题

4. **垂直切片比水平切片更好并行化**
   - 垂直：端到端功能（Plan 1: 用户功能，Plan 2: 产品功能）
   - 水平：按层（Plan 1: 所有模型，Plan 2: 所有 API）
   - 原因：层间依赖多，垂直切片减少依赖

### 方法论洞察

5. **复杂性封装，而非暴露**
   - 系统内部：复杂（上下文工程、XML 格式化、子智能体编排、状态管理）
   - 用户接口：简单（5 个核心命令）
   - 启示：好的工具应该隐藏复杂性，而非增加复杂性

6. **验证是不可或缺的**
   - 自动化验证：代码存在、测试通过
   - 人工验证：功能符合预期
   - 失败诊断：自动生成修复计划

7. **记忆是跨会话连续性的关键**
   - STATE.md：决策、阻塞、位置
   - todos/：捕获的想法
   - threads/：跨会话上下文
   - seeds/：前瞻性想法

8. **灵活性是系统设计的核心**
   - 可调整粒度（coarse/standard/fine）
   - 可切换模式（discuss/assumptions）
   - 可配置模型（quality/balanced/budget）
   - 可插入技能（agent_skills）

### 业务洞察

9. **solo 开发者 ≠ 小能力**
   - GSD 让 solo 开发者拥有企业级的系统化能力
   - 无需假装是 50 人公司
   - 专注于构建，而非流程

10. **AI 辅助编程需要新的工作流**
    - 不是"替代开发者"，而是"增强开发者"
    - 不是"生成代码"，而是"可靠地构建系统"
    - 关键：上下文工程 + 结构化流程

11. **社区和生态系统的重要性**
    - 多运行时支持：Claude Code, OpenCode, Gemini, Codex, Copilot, Cursor, Windsurf, Antigravity
    - 活跃社区：Discord, GitHub 讨论
    - 持续更新：快速迭代（41 个 releases）
