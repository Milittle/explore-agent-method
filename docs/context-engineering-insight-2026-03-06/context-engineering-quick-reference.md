# 上下文工程 - 快速参考指南

## 一句话总结

**上下文工程是管理进入LLM的所有信息（系统提示、工具、文档、历史、输出）的学科，核心是找到实现目标所需的最小高信号token集合。**

---

## 核心原则速查表

| 原则 | 一句话说明 | 立即行动 |
|------|------------|----------|
| **渐进式披露** | 按需加载，不一次性加载所有信息 | 创建三层结构（摘要→模块→详情） |
| **上下文隔离** | 子智能体用于分割上下文，非角色模拟 | 给每个子任务独立上下文窗口 |
| **工具整合** | 人类都选不对的工具，AI也选不对 | 合并相关工具为一个综合工具 |
| **质量>数量** | 上下文越长≠效果越好 | 只包含对当前决策有影响的信息 |
| **时间追踪** | 事实会变，不追踪会污染上下文 | 记录valid_from和valid_until |
| **结果导向评估** | 关注目标达成，不关心具体路径 | 用多维评分标准，不检查步骤 |

## 常见问题快速修复

| 问题 | 快速修复 | 参考技能 |
|------|----------|----------|
| 上下文太长，成本高 | 渐进式披露+观察掩码 | context-fundamentals |
| 智能体选错工具 | 合并重叠工具，改进描述 | tool-design |
| 长任务性能下降 | 检查点+压缩+多智能体 | context-compression |
| 记忆混乱/过时 | 添加时间有效性追踪 | memory-systems |
| 多智能体协调混乱 | 实施forward_message工具 | multi-agent-patterns |
| 质量不稳定 | 构建多维评估框架 | evaluation |

## 架构选择决策树

```
是否需要AI处理？
├─ 否 → 用传统代码
└─ 是 → 是否需要跨会话记忆？
    ├─ 否 → 单智能体
    └─ 是 → 任务是否超出单一上下文限制？
        ├─ 否 → 单智能体+记忆系统
        └─ 是 → 任务是否有清晰阶段？
            ├─ 是 → 监督者架构
            └─ 否 → P2P架构
```

## 快速实施模板

### 模板1: 三层加载结构
```yaml
# L1: 总是加载 (~50 tokens)
system_prompt: 核心指令
tools: 工具名称和摘要

# L2: 任务触发时加载 (~200-500 tokens)
module_instructions: 模块详细说明
templates: 相关模板

# L3: 需要时加载
data_files: 具体数据
history: 历史记录
references: 参考文档
```

### 模板2: 综合工具描述
```python
def comprehensive_tool(
    action: Literal["action1", "action2", "action3"],
    params: Dict
) -> Dict:
    """
    统一工具名称

    使用场景:
    - 明确说明何时使用这个工具
    - 列出触发条件

    操作:
    - action1: 做什么
    - action2: 做什么
    - action3: 做什么

    返回:
    - concise: 简洁格式包含什么
    - detailed: 详细格式包含什么

    错误处理:
    - ERROR_TYPE: 如何恢复
    """
```
